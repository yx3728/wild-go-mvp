# Prototype Instructions

Run the local server yourself and open the preview in the in-app browser. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Design Context

- Wild Go is a mobile product interface for ordinary city explorers, not a scientific observation database or a marketing site.
- `docs/card-visuals/` contains the canonical visual references for capture, binder, and friends/showcase surfaces.
- Every photographed organism becomes a collectible card. Rarity must be legible on the card face through stars, material, label, and color, never color alone.
- The rarity ceiling is six stars. Six-star cards are holographic chase cards; lower tiers remain tactile and collectible without competing with the top tier.
- Cards should feel physically interactive through tilt, press, flip, and drag/showcase behaviors, with reduced-motion support.
- App rarity means discovery difficulty, not conservation status. Sensitive locations stay softened and wildlife-safe guidance remains visible.
- Preserve the restrained botanical product shell; reserve prismatic effects for rare card materials and reveal moments.
